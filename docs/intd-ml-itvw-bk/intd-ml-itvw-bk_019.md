# 1.1.3.4 机器学习生产中的其他技术角色

> 原文：[`huyenchip.com/ml-interviews-book/contents/1.1.3.4-other-technical-roles-in-ml-production.html`](https://huyenchip.com/ml-interviews-book/contents/1.1.3.4-other-technical-roles-in-ml-production.html)

机器学习生态系统中还有许多其他技术角色。其中许多角色根本不需要机器学习知识，例如，你可以为机器学习构建工具和基础设施，而不必知道神经网络是什么（尽管了解可能有助于你的工作）。例如，包括在 NVIDIA 工作的框架工程师，他们致力于 CUDA 优化，或者在 Google 工作的 TensorFlow 工程师，或者在 Amazon 工作的 AWS 平台工程师。

**机器学习基础设施工程师，机器学习平台工程师**

由于机器学习资源密集，它依赖于可扩展的基础设施。拥有成熟机器学习管道的公司通常有基础设施团队来帮助他们构建机器学习的基础设施。对于机器学习基础设施/平台工程师来说，宝贵的技能包括熟悉并行性、分布式计算和底层优化。

这些技能难以学习，需要时间来掌握，因此公司更倾向于雇佣已经在这方面有技能的工程师，并在机器学习方面对他们进行培训。如果你是那种既了解系统又了解机器学习的稀有品种，你将会很受欢迎。

**机器学习加速器/硬件工程师**

硬件是机器学习的主要瓶颈。许多机器学习算法受到处理器的限制，无法快速进行计算，没有足够的内存来存储数据/模型并将它们加载到内存中，成本不够低廉以进行大规模实验，以及没有足够的电力在设备上运行应用程序**在设备上**。

专注于为训练和部署机器学习模型构建硬件的公司数量激增，这些硬件公司既适用于云计算也适用于边缘计算。这些硬件公司需要具有机器学习专业知识的人来指导他们的处理器开发过程：决定要关注哪些机器学习模型，然后在其硬件上实施、优化和基准测试这些模型。越来越多的硬件公司也在探索使用机器学习算法来改进他们的芯片设计过程^(11)^(12)。

**机器学习解决方案架构师**

这个角色通常出现在为使用机器学习的其他公司提供服务和/或产品的公司中。由于每个公司都有自己的用例和独特的要求，这个角色涉及与现有或潜在客户合作，以确定你的服务和/或产品是否可以帮助他们的用例，如果是的话，如何帮助。

**开发者倡导者，开发者计划工程师**

你可能见过针对机器学习（ML）的开发者关系（devrel）工程师角色，如**开发者倡导者**和**开发者计划工程师**。这些角色在构建 ML 产品和使用这些产品的开发者之间架起沟通的桥梁。具体职责因公司而异，因角色而异，但你可以期待它们是成为 ML 产品的首批用户、编写教程、发表演讲、收集并回应社区反馈的组合。像 TensorFlow 和 AWS 这样的产品，其部分受欢迎程度归功于他们优秀的 devrel 工程师不懈的努力。

以前，这些角色仅见于大型公司。然而，随着许多机器学习初创公司现在遵循[开源核心商业模式](https://en.wikipedia.org/wiki/Open-core_model)——开源其核心或功能受限版本的产品，同时提供商业版本作为专有软件——这些初创公司需要与开发者社区建立并维护良好的关系，devrel 角色对于他们的成功至关重要。这些角色通常很难填补，因为很难找到既优秀又擅长沟通的工程师。如果你是一名希望与社区有更多互动的工程师，你可能想要考虑这个选项。
